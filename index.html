Teaching Philosophy

An Evidence-Based Approach to Software Engineering Education

My teaching practice is grounded in established UK higher education pedagogy, adapted for programming education and a globally distributed student community.

Constructive Alignment

I design modules backwards from clearly stated learning outcomes, ensuring lectures, live-coding labs, and summative assessment all measure the same intended competencies — so students always know what a good piece of Java or Python code looks like and why each exercise matters.

Learning to Code by Building

I favour authentic, project-based learning over isolated exercises: students build a working system incrementally across a module — from a single class to a tested, version-controlled application — mirroring how software is actually engineered in industry.

Inclusive & Accessible Distance Learning

Materials are built for a diverse, asynchronous-first cohort: captioned live-coding recordings, structured weekly labs, flexible engagement windows, and plain-language instructions that remove unnecessary barriers for students studying around work and family commitments.

Assessment as Dialogue

I treat feedback as an ongoing conversation rather than a final verdict: code-review-style formative checkpoints, rubric transparency, and feed-forward comments give students a clear route to improve their code before — not just after — summative submission.

Responsible AI-Assisted Development

I introduce Generative AI and code-completion tools (e.g. GitHub Copilot, ChatGPT) as tools every working software engineer now uses, within clear academic integrity boundaries — teaching students to critically review, test, and take responsibility for AI-suggested code rather than to copy it blindly.

Assessment & Feedback

Assessment Strategy

An authentic, portfolio-based assessment design — no unseen examination — built around real code, testing, and version control, and aligned with industry-relevant practice.

Coursework 1 — Individual OOP Portfolio

40% · Tests LO1–LO4

Individual Java (or Python) submission implementing a small object-oriented system from a supplied brief, including UML design diagrams, unit tests, and a 1,000-word reflective commentary justifying design and testing choices.

Coursework 2 — Group Agile Software Project

60% · Tests LO2–LO5

Teams design, build, and test a database-integrated Java or Python application over three sprints, using GitHub for version control and a Scrum-style backlog, submitting sprint reports, a technical report, and a working, tested codebase.

Sample Feedback

Formative Feedback — Week 6 Code Review Clinic

"Your class hierarchy is clean and your use of the Strategy pattern is a good fit for this problem. To strengthen your design before submission, add a unit test for the edge case where the input list is empty, and consider extracting that duplicated validation logic into a shared method."

Summative Feedback — Coursework 2 (66%, 2:1)

"This is a 2:1 standard submission. Your Git history (LO5) shows genuinely iterative, sprint-based development, and your JUnit test suite (LO4) covers the core service layer well. To move into First-class territory, extend test coverage to your database access layer and give a more critical discussion of alternative designs considered in your technical report."

Distance Learning & AI in Education

Online Pedagogy & AI Integration

Designing for a globally distributed, time-zone-diverse student body requires deliberate structure, accessibility, and the responsible use of new technologies.

Distance Learning Strategy

  • Synchronous Zoom seminars with live coding demonstrations, polls, and breakout-room collaboration
  • Asynchronous recorded lectures, captioned for accessibility, paired with structured weekly VLE content
  • Discussion boards and peer-review forums to sustain engagement between live sessions
  • Flexible office hours and 1:1 tutorial slots to support students across time zones
  • Inclusive, WCAG-aware materials and flexible deadlines for diverse and part-time learners
  • Use of LMS platforms (Moodle / Blackboard) for consistent, navigable module delivery

AI-Assisted Development in Teaching & Learning

  • Introduce Generative AI and code-completion tools (e.g. GitHub Copilot, ChatGPT) as a study aid for formative practice and code-explanation, within clear academic-integrity guidelines
  • Teach AI literacy explicitly — when and how to use AI coding assistants responsibly in coursework and future employment as a software engineer
  • Safeguard assessment validity through process-based and authentic tasks (Git commit history, live coding vivas, reflective commentaries) that are resistant to unattributed AI use
  • Use AI tools to draft formative practice exercises, accessibility adaptations, and rubric-aligned feedback templates, always with human academic review
  • Model critical, evaluative engagement with AI-suggested code — consistent with my own research into Explainable AI
